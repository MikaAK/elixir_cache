# elixir_cache Library Reference

Source: https://github.com/MikaAK/elixir_cache

The goal of this library is to unify Cache APIs and make adapters easy to implement and sharable across all storage types. Testing is a first-class concern — caches are isolated per test and don't leak state.

## Cache Behaviour

```elixir
@callback child_spec({cache_name :: atom, cache_opts :: Keyword.t()}) :: Supervisor.child_spec()
@callback opts_definition() :: Keyword.t()
@callback start_link(cache_opts :: Keyword.t()) :: {:ok, pid()} | {:error, term()} | :ignore
@callback put(cache_name :: atom, key :: atom | String.t(), ttl :: pos_integer, value :: any) :: :ok | ErrorMessage.t()
@callback put(cache_name :: atom, key :: atom | String.t(), ttl :: pos_integer, value :: any, Keyword.t()) :: :ok | ErrorMessage.t()
@callback get(cache_name :: atom, key :: atom | String.t()) :: ErrorMessage.t_res(any)
@callback get(cache_name :: atom, key :: atom | String.t(), Keyword.t()) :: ErrorMessage.t_res(any)
@callback delete(cache_name :: atom, key :: atom | String.t(), opts :: Keyword.t()) :: :ok | ErrorMessage.t()
@callback delete(cache_name :: atom, key :: atom | String.t()) :: :ok | ErrorMessage.t()
```

## Available Adapters

| Adapter | Description |
|---------|-------------|
| `Cache.Agent` | Simple agent-based caching |
| `Cache.DETS` | Disk-persisted caching with Erlang DETS |
| `Cache.ETS` | High-performance in-memory cache with ETS |
| `Cache.Redis` | Redis adapter using Redix & Poolboy, supports Redis JSON and Redis Hashes |
| `Cache.ConCache` | Wrapper around the ConCache library |
| `Cache.Sandbox` | Test adapter (auto-selected when `sandbox?: true`) |

Some adapters inject adapter-specific functions into your cache module via `__using__/1`. For example, `Cache.Redis` adds hash and pipeline functions, and `Cache.ETS` adds the full set of ETS operations listed below.

## `use Cache` Macro

When you `use Cache`, the following is generated:

**Required options:**
- `adapter` — one of the adapters above (required)
- `name` — atom cache name (required)

**Optional:**
- `sandbox?` — if true, uses `Cache.Sandbox` adapter for test isolation
- `opts` — adapter options, can be:
  - Keyword list (validated against adapter's `opts_definition/0`)
  - `{module, function, args}` — MFA tuple
  - `{application_name, key}` — Application env lookup
  - `application_name` — Application env lookup with module as key
  - Zero-arity function

**Generated functions:**
- `cache_name/0` — returns the cache name atom
- `cache_adapter/0` — returns the adapter module
- `adapter_options/0` — resolves and returns adapter options
- `child_spec/1` — for supervision tree
- `put/2`, `put/3` — store value with optional TTL
- `get/1` — retrieve value (returns `{:ok, value}` or `{:ok, nil}`)
- `delete/1` — remove key
- `get_or_create/2` — get or compute and store

Values are automatically encoded/decoded via `Cache.TermEncoder` (binary term format with optional compression).

## Starting Caches

In your `application.ex`:

```elixir
children = [
  {Cache, [MyModule]}
]
```

## Runtime Configuration

Adapter options can be resolved at runtime:

```elixir
# MFA tuple
use Cache, adapter: Cache.Redis, name: :my_cache, opts: {MyModule, :opts, []}

# Zero-arity function
use Cache, adapter: Cache.Redis, name: :my_cache, opts: &MyModule.opts/0

# Application config: Application.fetch_env!(:my_app, MyModule)
use Cache, adapter: Cache.Redis, name: :my_cache, opts: :my_app

# Application config with key: Application.fetch_env!(:my_app, :cache_opts)
use Cache, adapter: Cache.Redis, name: :my_cache, opts: {:my_app, :cache_opts}
```

## Cache.ETS Adapter

**Options** (NimbleOptions):
- `write_concurrency: boolean | :auto` — enable write concurrency (`:auto` supported)
- `read_concurrency: boolean` — enable read concurrency
- `decentralized_counters: boolean` — use decentralized counters
- `type: :set | :bag | :duplicate_bag` (default `:set`)
- `compressed: boolean` — enable ETS compression
- `rehydration_path: string` — persist ETS to disk on exit and rehydrate on startup

**Error handling:** The ETS adapter's `get/3`, `put/5`, and `delete/3` all rescue exceptions and return `{:error, ErrorMessage.internal_server_error(...)}` instead of crashing.

**Additional ETS functions** generated by `use Cache` with ETS adapter:
- `all/0`, `delete_table/0`, `delete_all_objects/0`, `delete_object/1`
- `first/0`, `last/0`, `next/1`, `prev/1`
- `first_lookup/0`, `last_lookup/0`, `next_lookup/1`, `prev_lookup/1` (OTP 26+)
- `lookup/1`, `lookup_element/2`, `lookup_element/3` (OTP 26+), `member/1`
- `insert_raw/1`, `insert_new/1`
- `match_pattern/1`, `match_pattern/2`, `match_delete/1`
- `match_object/1`, `match_object/2`, `match/1`
- `match_spec_compile/1`, `match_spec_run/2`
- `select/1`, `select/2`, `select_count/1`, `select_delete/1`, `select_replace/1`
- `select_reverse/1`, `select_reverse/2`
- `foldl/2`, `foldr/2`
- `update_counter/2`, `update_counter/3`, `update_element/2`, `update_element/3` (OTP 26+)
- `tab2list/0`, `tab2file/1`, `tab2file/2`, `tabfile_info/1`, `file2tab/1`, `file2tab/2`
- `info/0`, `info/1`, `table/0`, `table/1`, `take/1`, `whereis/0`
- `to_dets/1`, `from_dets/1`
- `safe_fixtable/1`, `rename/1`, `give_away/2`, `setopts/1`, `slot/1`
- `init_table/1`, `repair_continuation/2`, `test_ms/2`, `is_compiled_ms/1`

**ETS rehydration:** When `rehydration_path` is set, the adapter traps exit signals (SIGTERM, SIGHUP, SIGQUIT, etc.) and writes the table to disk via `tab2file`. On startup, it attempts to rehydrate from the file. If the file has a different table name or is corrupt, a fresh table is created.

## Cache.get_or_create/3

```elixir
Cache.get_or_create(MyCache, "user:123", fn ->
  case UserRepo.get(123) do
    nil -> {:error, ErrorMessage.not_found("User not found")}
    user -> {:ok, user}
  end
end)
```

Pattern: get from cache → if nil, call function → if `{:ok, value}`, store and return.

## Telemetry Events

- `[:elixir_cache, :cache, :put]` — span
- `[:elixir_cache, :cache, :get]` — span
- `[:elixir_cache, :cache, :get, :miss]` — counter on cache miss
- `[:elixir_cache, :cache, :delete]` — span
- `[:elixir_cache, :cache, :put, :error]` — counter on error
- `[:elixir_cache, :cache, :get, :error]` — counter on error
- `[:elixir_cache, :cache, :delete, :error]` — counter on error

All events include `%{cache_name: cache_name}` metadata.

## Sandbox Testing

When `sandbox?: true`:
- The `Cache.Sandbox` adapter is used automatically (a simple Agent cache unique to the root process)
- Keys are prefixed with a sandbox ID from `Cache.SandboxRegistry`
- Each test process gets isolated cache keys

**Setup required** in `test/test_helper.exs`:

```elixir
Cache.SandboxRegistry.start_link()
ExUnit.start()
```

Then in each test's `setup`:

```elixir
Cache.SandboxRegistry.start([MyCache, OtherCache])
```

## Creating Custom Adapters

Implement the `@behaviour Cache` callbacks:

```elixir
put(cache_name, key, ttl, value, opts \\ [])
get(cache_name, key, opts \\ [])
delete(cache_name, key, opts \\ [])
opts_definition()  # NimbleOptions definition
child_spec({cache_name, cache_opts})
```

If the adapter exports `__using__/1`, it will be invoked to inject adapter-specific functions into the cache module.
